## 比较介绍

这是我们在 Blue 的第一个比较页面。

这将与您在互联网上阅读的所有其他比较页面不同。与许多在线比较页面不同，这不是一个掩饰的广告。您可以查看 [Clickup 的 Trello 比较页面](https://clickup.com/compare/trello-vs-clickup) 来获取一个荒谬的例子。

我们不是来抨击竞争对手或声称在各个方面都优于对方。

这将是对 Blue 和 Trello 之间差异和相似之处的*诚实*审视，以便**您**可以决定什么最适合您的需求，以及 Blue 是否可以成为 Trello 的替代品。

事实是，没有*最佳*的项目管理和协作软件适合所有组织——人们的思维和工作方式差异太大，无法得出这样的结论。

最终，我们的目标是提供一个平衡的视角，尊重两个平台的优势，同时提供您做出明智决策所需的信息。

无论您是寻找简单的小团队，还是需要高级功能的大型企业，这个比较将帮助您确定哪个工具最符合您的独特需求。

## Trello 介绍

[Trello](https://trello.com) 是项目管理软件的开创者之一——它于 2011 年 9 月推出！

它由 Fog Creek Software（现在的 [Glitch](https://glitch.com/)）的联合创始人 Joel Spolsky 和 Michael Pryor 创建。

Joel 有一个很棒的博客，名为 [Joel on Software](https://www.joelonsoftware.com/)。

- 无痛功能规格（[第一部分](https://www.joelonsoftware.com/2000/10/02/painless-functional-specifications-part-1-why-bother/)，[第二部分](https://www.joelonsoftware.com/2000/10/03/painless-functional-specifications-part-2-whats-a-spec/)，[第三部分](https://www.joelonsoftware.com/2000/10/04/painless-functional-specifications-part-3-but-how/)，[第四部分](https://www.joelonsoftware.com/2000/10/15/painless-functional-specifications-part-4-tips/)）

老实说，如果您与软件开发有关，请立即暂停，打开每一篇文章并在新标签页中书签以便稍后阅读。

这简直是纯金。

好了——回到 Trello！

Trello 的开发旨在提供一种简单而直观的方式来管理项目和组织任务，使用板、列表和卡片的系统。

在 2011 年推出时，网络技术还不是今天的样子。我们今天认为理所当然的许多技术和技巧尚未被发明。

像 [React](https://reactjs.org/)、[Angular](https://angular.io/) 和 [Vue.js](https://vuejs.org/)（Blue 使用的框架）这样的 JavaScript 框架尚未推出，它们彻底改变了前端开发，开发者依赖于较旧的工具，如 [Backbone.js](https://backbonejs.org/) 或 [jQuery](https://jquery.com/)。 [WebAssembly (Wasm)](https://webassembly.org/)，使 Web 应用程序接近本地性能，仍然需要数年才能实现。[Service Workers](https://developer.mozilla.org/en-US/docs/Web/API/Service_Worker_API) 和 [渐进式 Web 应用 (PWAs)](https://web.dev/progressive-web-apps/)，提供离线功能和后台同步，尚不可用，因为它们是在 2014 年推出的。[CSS 网格布局](https://developer.mozilla.org/en-US/docs/Web/CSS/CSS_Grid_Layout) 和 [Flexbox](https://developer.mozilla.org/en-US/docs/Web/CSS/CSS_Flexible_Box_Layout/Basic_Concepts_of_Flexbox)，提供高级布局选项，要么不受支持，要么未完全采用。[Web 组件](https://developer.mozilla.org/en-US/docs/Web/Web_Components)，允许可重用的自定义元素，尚未得到广泛支持。[GraphQL](https://graphql.org/)，提供 REST API 的灵活替代方案，以及现代构建工具如 [Webpack](https://webpack.js.org/) 和包管理器如 [Yarn](https://yarnpkg.com/)，尚未出现。[HTTP/2](https://http2.github.io/)，显著提升 Web 性能，直到 2015 年才标准化。

Trello 的开发必须克服这些技术限制。

使用可视化板来灵活、直观地组织任务和项目的概念是突破性的，尤其是在没有现代 Web 技术的情况下。Trello 利用当时 Web 开发的进步，如 AJAX 和实时更新，创造了无缝和互动的用户体验。

这种方法显著改善了传统项目管理工具，后者通常更为僵化且不够直观。Trello 的成功证明了基于 Web 的应用程序可以为复杂任务提供复杂且用户友好的解决方案的潜力。

Trello 迅速发展，并在 2014 年获得了由 [Index Ventures](https://www.indexventures.com/) 和 [Spark Capital](https://www.sparkcapital.com/) 领导的 1030 万美元 A 轮融资，随后于 2017 年 1 月以 4.25 亿美元被 [Atlassian](https://atlassian.com) 收购。此次收购包括约 3.6 亿美元现金和其余的 Atlassian 股票。

## Blue 介绍

Blue 于 2018 年由 [Emanuele Faja](https://www.linkedin.com/in/efaja/) 创立，使命明确：

> 通过构建地球上最好的项目管理平台来组织世界的工作——简单、强大、灵活且对所有人都负担得起。

为了让事情更清晰，Trello 在 Blue 开始之前就*已经*被卖给了 Atlassian！

因此，自然，这两个平台在项目管理的许多方面将采取不同的方法。

与许多科技初创公司不同，Blue 自豪地采用自筹资金的方式，*没有*风险投资资金的支持。这种方法使 Blue 能够保持对其愿景的忠诚，并保持对用户需求和产品质量的强烈关注，而不必在投资者需求与客户需求之间权衡。

Blue 的灵感来自于创建一个不仅易于使用，而且足够灵活和强大以处理最苛刻工作负载的工具的愿望。实质上，是对像 Trello 和 Basecamp 这样的软件的更好替代品，它们易于使用，但可能缺乏处理更复杂工作负载的能力。

Blue 的创始人认识到需要一种将简单性与高级功能相结合的项目管理解决方案，满足从小团队到大型企业的广泛用户需求。这种在可用性与稳健性之间保持平衡的承诺使 Blue 在竞争激烈的项目管理软件市场中脱颖而出。

## 自动化限制

让我们从 [自动化](/platform/features/automations) 开始。

Blue 和 Trello 都拥有复杂的自动化引擎。[Trello 将其自动化引擎](https://trello.com/butler-automation) 称为“Butler”，我们认为这是一个相当酷的名字。

然而，[Trello 确实根据您的计划和用户数量强制执行配额](https://support.atlassian.com/trello/docs/butler-quotas-and-limits/)：

| 特性                     |        免费        |      标准      |          高级          |        企业        |
|--------------------------|:------------------:|:--------------:|:----------------------:|:------------------:|
| 每月自动化运行次数       |        250         |      1,000     |        无限            |        无限        |
| 每月操作次数             |      2,500         |      20,000    | 每个付费用户 10,000 次，限额为 250,000 | 每个用户 10,000 次 |

在这里值得区分的是“自动化运行”和“操作”：

- **自动化运行：** 自动化运行是自动化命令的单次执行。例如，如果您有一个规则，当到期日到达时将卡片移动到另一个列表，每次触发该规则时，它都算作一次自动化运行。
- **操作：** 操作是作为自动化运行的一部分执行的单个操作。例如，如果一个规则移动一张卡片（一个操作），然后为该卡片添加一个到期日（另一个操作），那么这将算作在一次自动化运行中的两个操作。

鉴于这些限制，**如果您计划进行大量自动化而用户不多，Blue 可能是最佳选择**，因为我们对自动化的数量没有任何限制。

这意味着您可以创建任意数量的自动化规则，而无需担心达到上限，使 Blue 成为高度依赖自动化的小团队的理想选择。

然而，Trello 确实有一些 Blue 尚未具备的独特功能：

- **条件自动化：** Trello 的 Butler 自动化引擎支持创建根据特定条件执行操作的规则。例如，您可以设置一个规则，仅在卡片具有某个标签、位于特定列表中或满足您定义的其他条件时触发。这些条件自动化允许在 Trello 中实现更个性化和有针对性的工作流程。
- **与第三方应用的自动化：** Trello 允许通过其 Butler 自动化引擎与各种第三方应用集成。这意味着您不仅可以在 Trello 内部自动化操作，还可以跨越您的团队可能使用的其他工具和服务，从而增强项目管理系统的整体工作流程和连接性。Blue 仅通过 [Webhooks](https://documentation.blue.cc/integrations/webhooks) 和第三方集成商如 [Pabbly](https://documentation.blue.cc/integrations/pabbly-connect) 和 [Zapier](https://documentation.blue.cc/integrations/zapier) 支持此功能。

Blue 具有 [强大的电子邮件自动化功能](/insights/email-automations)，这是 Trello 所不具备的，允许您向任何电子邮件地址发送自定义电子邮件通知，利用您 [自定义字段](https://www.blue.cc/platform/features/custom-fields) 中的数据，使用动态标签。这对于 [管理服务票据](/solutions/use-case/service-tickets) 等工作流程非常有用。

## 可扩展性

[Trello 文档](https://developer.atlassian.com/cloud/trello/guides/rest-api/limits/#board-level-limits) 规定每个板允许的最大卡片数为 5,000，整个工作区的总数（包括归档）为 2,000,000。

Blue 的可扩展性显著更强，允许每个列表最多 100,000 条记录（相当于 Trello 卡片）。我们的 CSV 导入允许您一次导入多达 250,000 条记录——[您可以在我们的工程博客上阅读我们是如何实现这一点的。](/insights/scaling-csv-import-export)

因此，如果您希望在项目中处理大量数据，Blue 是一个很好的 Trello 替代品。

## 视图比较

Trello 和 Blue 有许多不同的方式来查看相同的数据集。

在 Blue 中，所有客户都可以使用所有视图，而在 Trello 中，免费和标准计划仅具有 [看板视图](/platform/features/kanban-board)，其他视图仅在其更昂贵的高级（每用户 12.50 美元/月）和企业（每用户 17.50 美元/月）计划中可用。

这就是为什么在定价比较部分将 Blue 与 Trello 高级和企业进行比较——这是基于功能集和能力的唯一真实比较。

| 视图                     | Trello                            | Blue |
|--------------------------|:---------------------------------:|:----:|
| 板视图                   |   ✅                              |  ✅  |
| 时间线视图              |   ✅（高级和企业）                |  ✅  |
| 日历视图                 |   ✅（高级和企业）                |  ✅  |
| 仪表板视图               |   ✅（高级和企业）                |  ✅  |
| 表格视图                 |   ✅（高级和企业）                |  ✅  |
| 地图视图                 |   ✅（高级和企业）                |  ✅  |
| 跨项目日历              |   ✅（高级和企业）                |  ✅  |
| 跨项目表格              |   ✅（高级和企业）                |  ❌  |
| 项目活动                 |   ✅                             |  ✅  |
| 跨项目活动              |   ❌                              |  ✅  |

Blue 缺少跨项目表格视图，但 Trello 也没有跨项目活动视图，您可以查看所有卡片/记录中谁做了什么。

Blue 还具有 [锁定整个列表](https://documentation.blue.cc/records/lists#list-locking) 的能力，这会阻止以下操作：

- **重命名：** 一旦锁定，列表名称无法更改。这保持了命名的一致性。
- **批量操作：** 批量记录操作，如“全部完成”、“全部分配”、批量标记等，被禁用。
- **移动记录：** 现有记录无法移入或移出锁定列表。
- **创建记录：** 新记录无法直接添加到锁定列表。
- **删除：** 锁定列表无法被完全删除。

许多 Trello 用户（例如 [这里](https://community.atlassian.com/t5/Trello-questions/Can-you-lock-certain-lists-on-a-Trello-board-to-not-be-editable/qaq-p/1322995) 和 [这里](https://community.atlassian.com/t5/Trello-questions/How-to-lock-a-list-in-place-on-the-board/qaq-p/2309184)）已经请求此功能，但 Trello 尚未实现。

因此，如果您需要锁定工作流程中特定部分的能力，那么 Blue 可能是 Trello 的一个很好的替代品。

## 用户权限

这是 Blue *真正* 突出的一个领域。

让我们看看 Trello 的用户角色：

* **板管理员**：可以在板上执行任何操作，包括更改权限。其头像右下角标有蓝色和白色的标记。
* **普通成员**：可以在没有限制的情况下编辑和添加到板上。
* **观察者**：对板的只读访问权限。可以查看和互动，但不能进行更改。仅在高级和企业计划中可用。
* **工作区成员**：如果他们是该工作区的成员，可以查看工作区可见的板。
* **访客**：是板的普通成员，但不是其所属工作区的成员。

Trello 用户报告说，“观察者”角色仅在高级和企业计划中可用，这令人烦恼。此外，[观察者角色只有在被邀请到一个板时才免费](https://support.atlassian.com/trello/docs/adding-observers-to-boards/)，如果您邀请他们到多个板，您必须像普通团队成员一样支付每用户费用。

在 Blue 中，您可以拥有无限数量的“仅评论”和“仅查看”用户，免费提供，这与 Trello 中的“观察者”角色相同。

此外，Trello 目前不允许管理员即时创建自定义用户角色。该平台提供了一组预定义的用户角色，具有特定权限，但不提供动态创建和配置新角色的能力。

Blue 既有默认角色，也允许项目管理员根据项目的具体需求创建*自定义*用户角色。以下是概述：

- **项目管理员**：对项目拥有完全控制权，包括管理设置、用户和权限。
- **团队成员**：可以以编辑和协作的能力为项目做出贡献。
- **客户**：有限的访问权限，可以查看和评论特定项目元素。
- **仅评论**：可以查看和添加评论，但不能编辑项目内容。
- **仅查看**：对项目信息的只读访问权限，没有编辑能力。
- **自定义用户角色**：根据项目内的特定需求和权限量身定制的可自定义角色。

有关更多详细信息，请访问 [Blue 用户角色文档](https://documentation.blue.cc/user-management/roles)。

通过自定义用户角色，您可以精确配置用户应看到和执行的内容。

在 Blue 中，您可以轻松地根据团队的需求自定义用户权限。这种灵活性确保每个人都能访问正确的工具和信息。例如，通过基于组的权限，您可以创建适用于整个团队的角色，这有助于保持具有相似任务的成员的一致访问权限。这是一种保持项目工作流程有序和高效的简单方法。

自定义角色让您控制谁可以访问各种活动和表单，保护敏感信息的安全，同时确保您的团队专注于正确的任务。您甚至可以调整可见性设置，以便组仅查看与他们相关的记录。这不仅简化了他们的工作流程，还帮助他们保持专注。

此外，您可以将邀请权限分配给特定角色。这允许选定用户邀请新成员加入他们的组，使您能够在不失去对安全性或结构控制的情况下轻松扩展团队。这对于希望顺利增加新成员的成长型团队特别方便。

另一个有益的功能是自定义字段的管理。项目管理员可以设置查看或编辑特定字段的权限，这有助于维护数据的完整性和实用性。无论是简单的文本字段还是复杂的数据，如 URL、文档、设计或时间持续，您都拥有所需的灵活性。

通过自定义角色，管理列表也更加高效。您可以指定谁可以查看、编辑或删除项目中的列表。这种控制水平为高效管理项目创造了一个量身定制和安全的环境。

因此，如果您正在寻找团队的细粒度权限，那么 Blue 是一个很好的 Trello 替代品，因为 Trello 的自定义选项有限。

## 任务依赖关系

在管理任务依赖关系方面，Blue 提供了 Trello 所缺乏的强大功能，使其成为一个很好的替代品。

**Trello 不原生支持任务依赖关系**，这意味着您无法直接链接任务以创建结构化的工作流程，其中一个任务的完成依赖于另一个任务。这一限制可能会妨碍项目管理的效率，特别是对于需要顺序操作的复杂任务。您可以购买 Power-Ups 来额外收费，但这些是必须管理的第三方集成，如果 Trello 中发生变化，可能会中断。

相比之下，Blue 在这一领域表现出色，全面*原生*支持任务依赖关系。Blue 允许您连接记录以创建结构化工作流程，确保任务按要求的顺序完成。

在 Blue 中有两种类型的依赖关系：

1. **阻塞**：阻塞依赖关系意味着一个记录在完成之前阻止其他记录的启动或完成。
2. **被阻塞**：被阻塞依赖关系表示一个记录被另一个必须先完成的记录阻止。

此功能有助于维护适当的任务顺序，通过在依赖关系未满足时显示警告，防止顺序错误的完成。此外，所有依赖关系都记录在活动部分，提供工作流程连接的透明度。

Blue 还包括 [引用](https://documentation.blue.cc/custom-fields/reference) 和 [查找](https://documentation.blue.cc/custom-fields/lookup) 自定义字段，进一步增强项目管理，允许跨不同项目的关系和数据可见性。引用自定义字段在不同项目的记录之间创建关系，而查找自定义字段从其他项目的记录中导入数据，确保所有必要信息可访问且最新。这些功能使 Blue 成为管理复杂工作流程和确保任务按正确顺序完成的强大工具。

## 文件存储

Blue 和 Trello 都为每个帐户提供慷慨的文件存储。然而，主要区别在于每个单独文件的大小限制。

Trello 在其计划中对此进行了不同的规定：

| 计划            | 每个文件附件限制 |
|-----------------|:-----------------:|
| 免费            | 10 MB             |
| 标准            | 250 MB            |
| 高级            | 250 MB            |
| 企业            | 250 MB            |

[Blue 提供的每个文件附件限制是 Trello 的 20 倍](/platform/features/file-management)，每个文件附件为 5GB。

这使得 Blue 成为上传视频、大型照片备份和其他大文件等任务的绝佳替代品。

此外，**Trello 不支持为文件附件创建公共共享链接。**

Blue 允许用户生成可共享链接，使与不使用 Blue 的客户或合作伙伴轻松共享文件。此功能增强了协作和可访问性，使您更容易在团队之外共享文件。

## APIs 和 Webhooks

### APIs
Trello 有一个 [文档齐全的 Rest API](https://developer.atlassian.com/cloud/trello/rest/api-group-actions/#api-group-actions)，使用起来非常简单。然而，存在一些限制：

- **高级自定义字段**：虽然 API 允许基本的卡片和板交互，但高级自定义字段并未完全可访问或可修改。
- **对 Power-Ups 的有限访问**：某些增强 Trello 功能的 Power-Ups 可能没有相应的 API 端点。这可能限制了以编程方式自动化或集成这些功能的能力。
- **用户界面元素**：API 不提供对所有用户界面元素或设置的访问，这些元素或设置可以在 Trello 应用中进行调整，例如特定的视觉自定义或布局选项。
- **实时协作功能**：使实时协作成为可能的功能，例如应用内的实时更新或通知，并未在 API 中完全复制。API 主要处理数据检索和操作，而不是实时交互。
- **速率限制**：API 对请求施加严格的速率限制（例如，API 密钥每 10 秒 300 次请求，令牌每 10 秒 100 次请求），这可能限制在短时间内可以访问或修改的数据频率和数量。

Blue 拥有一个 [GraphQL API，提供对整个平台的 100% 覆盖](https://www.blue.cc/platform/api)。

因此，您可以通过我们的接口执行的*任何*操作，都可以通过编程触发。这是我们在首次创建 API 时的一个有意识的设计选择，我们希望为开发者提供完全的访问权限。

### Webhooks

Trello 和 Blue 都有 Webhooks。Trello 的 Webhooks 是 [文档齐全的](https://developer.atlassian.com/cloud/trello/guides/rest-api/webhooks/)，但只能通过其 API 创建和监控。Trello 的 Webhook 功能主要集中在卡片创建、更新和删除等一般操作上。

另一方面，Blue 提供了程序化访问*和*创建、查看和管理 Webhooks 的 Web 界面。

Blue 的 Webhook 选项还提供了更细粒度的方法，允许用户选择特定事件来触发通知。这种级别的自定义包括对记录、检查表、自定义字段、标签和评论等的操作。这种细粒度确保您可以根据确切需求定制 Webhooks，仅在最相关的事件上触发。

此外，Blue 的 Webhooks 可以按一个或多个项目进行过滤，甚至跨不同的组织或工作区。这意味着您可以创建复杂的跨项目自动化规则和通知，增强各团队和部门之间的协调和生产力。通过用户友好的界面管理这些设置简化了过程，使即使没有编程专业知识的人也能轻松使用。

## 企业结构差异

在选择 [项目管理工具](/solutions/use-case/project-management) 时，考虑的不仅仅是功能。您还应该考虑产品背后的公司。这种更广泛的视角可以为您提供有关他们的优先事项、关注点和未来走向的深刻见解。

在 2017 年被 Atlassian 以 4.25 亿美元收购后，Trello 成为一家大型上市公司的组成部分。这种支持为 Trello 提供了访问一个以可靠性著称的成熟品牌的资源，尤其是与其无缝集成的 Jira 和 Confluence 等工具。

然而，成为大型组织的一部分也有其缺点。

Trello 在尝试实施用户反馈或快速更新时可能会遇到官僚主义挑战。此外，由于 Atlassian 拥有更昂贵的项目管理工具如 Jira，因此可能会存在利益冲突。他们可能会犹豫不决，不愿让 Trello 变得过于强大，以免遮蔽这些更高层次的产品。

或者，您可能会在 Trello 高级帐户中收到对 Jira 的烦人推销。

Atlassian 主要专注于满足大型企业的需求，这可能意味着他们的一些功能和定价并不真正适合中小型企业（SME）的需求。由于在各种产品中拥有如此庞大的用户基础，他们可能很难提供个性化支持或真正理解小型公司的独特需求。

Blue 是一家独立公司，专注于其项目管理平台。这种专注的方法使 Blue 能够更快地响应用户需求，并根据反馈进行调整。

作为一家中小型企业，Blue 理解小型企业面临的挑战。这就是为什么我们不对我们的计划进行功能限制，并且定价极具竞争力。我们的规模也意味着我们可以提供更个性化的支持，并与用户建立更强的关系。

由于 Blue 不受股东压力或大型公司的目标影响，我们可以专注于长期产品开发，而不是追求短期利润。这种专注确保我们关注用户需求，而不是企业战略。

因此，当您在 Trello 和 Blue 之间犹豫时，请考虑今天的功能集之外——考虑每个产品和公司未来可能的发展。

如果您是已经使用 Atlassian 产品的大型企业的一部分，您可能会非常欣赏 Trello 的集成功能。

另一方面，如果您是寻找真正理解您特定需求的工具的中小型企业，Blue 的专注方法可能更适合您。

此外，反思一下您需要从项目管理工具提供商那里获得什么样的支持。Trello 仅向付费客户提供支持票，并且仅向高级和企业层级提供 24/7 支持。

您在工具开发中有发言权的重要性有多大？您需要多快实施更改？

最终，最佳选择取决于您的独特需求、公司文化和长期商业目标。Trello 和 Blue 都提供可靠的项目管理解决方案，但它们不同的背景和关注点可能会影响它们未来如何为您服务。

## Trello 替代品？

总之，Trello 一直是管理项目的热门选择，但 Blue 具有一些令人印象深刻的优势，可能使其更适合您的需求。凭借更大的文件存储限制、易于使用的 API 和可自定义的用户权限，Blue 适合小团队和大型公司。此外，其清晰的定价——没有隐藏费用——使其成为一个预算友好的选择，特别适合正在增长的团队。

Blue 还具有高度可扩展性，提供无限自动化，并允许任务依赖关系，使其非常适合更复杂的项目。

选择 Blue，您选择了一种用户驱动的方法，实时适应您的需求。

如果您想要高级功能、更好的可扩展性，或者只是一个更具成本效益的工具，Blue 确实是 Trello 的一个出色替代品。
