ជាស្តីន រ៉ូសេនស្តីន បានធ្វើការជាអ្នកកម្មវិធីដឹកនាំនៅ Facebook ដែលគាត់បានណែនាំប៊ូតុង "ចូលចិត្ត"។ មុននឹងធ្វើការនៅ Facebook គាត់ក៏បានធ្វើការនៅ Google ជាអ្នកគ្រប់គ្រងផលិតផល ដែលបានចូលរួមក្នុងគម្រោងដែលក្រោយមកក្លាយជាប្រព័ន្ធ Google Drive និងគំរូ GChat ដំបូង។

អាសាណាបានកើតឡើងពីការមិនពេញចិត្តរបស់ពួកគេចំពោះការប្រព្រឹត្តទៅរបស់ភារកិច្ច និងការទំនាក់ទំនងដែលជាញឹកញាប់រារាំងផលិតភាពនៅក្នុងអង្គភាពធំៗ។ អាសាណាបានចំណាយប្រហែល ៤ ឆ្នាំចាប់ពីការបង្កើតនៅឆ្នាំ 2008 ដល់ការចេញផ្សាយពាណិជ្ជកម្មនៅឆ្នាំ 2012។ ក្រុមនេះមានគោលបំណងបង្កើតវេទិកាដែលនឹងធ្វើឱ្យការងារល្អប្រសើរឡើង និងកាត់បន្ថយពេលវេលាដែលចំណាយទៅលើភារកិច្ចគ្រប់គ្រង ដើម្បីអនុញ្ញាតឱ្យក្រុមអាចផ្តោតលើការងារសំខាន់របស់ពួកគេ។

ឆ្លងកាត់ទៅ ១៦ ឆ្នាំក្រោយ អាសាណាបានក្លាយជាក្រុមហ៊ុនដែលចុះបញ្ជីនៅសាធារណៈ ហើយបានជួញដូរ​នៅលើទីផ្សារមូលបត្រ New York។ 


## ការណែនាំអំពីប្លូ

ប្លូ ត្រូវបានបង្កើតឡើងនៅឆ្នាំ 2018 ដោយ [អេម៉ានូអែល "ម៉ានី" ហ្វាថា](https://www.linkedin.com/in/efaja/) ដែលមានគោលបំណងច្បាស់លាស់៖ 

> ដើម្បីរៀបចំការងារនៅលើពិភពលោក។

ប្លូ មានមោទនភាពក្នុងការបង្កើតដោយខ្លួនឯង។ 

យើងនៅក្នុងចំនួនតិចក្នុងឧស្សាហកម្មនេះ ដែលភាគច្រើននៃការប្រកួតបានបង្កើតប្រាក់រាប់រយលានដុល្លារនៅក្នុងការវិនិយោគ។ វិធីសាស្ត្រនេះបានអនុញ្ញាតឱ្យប្លូរក្សាទំនាក់ទំនងជាមួយវិស័យនិងរក្សាការផ្តោតលើតម្រូវការរបស់អ្នកប្រើ និងគុណភាពផលិតផល ជំនួសការប្រឈមមុខនឹងតម្រូវការរបស់អ្នកវិនិយោគជាមួយតម្រូវការរបស់អតិថិជន។ 

## ការប្រៀបធៀបតម្លៃ

ខណៈពេលដែលផែនការឥតគិតថ្លៃ ("ផ្ទាល់ខ្លួន") អាចបង្ហាញថាមានភាពទាក់ទាញ ប្រសិនបើអ្នកជាក្រុមតូចដែលមានអ្នកប្រើតិចជាង ១០ នាក់ អ្នកត្រូវតែគិតថាថាវាមានកំណត់ជាច្រើន។ 

វាអត់មានមុខងារគ្រប់គ្រងគម្រោងកម្រិតខ្ពស់ដូចជា ទិដ្ឋភាពពេលវេលា, តារាង Gantt, និងការពឹងផ្អែកលើភារកិច្ចដែលមាននៅក្នុងផែនការបង់ប្រាក់។ នេះធ្វើឱ្យវាពិបាកក្នុងការមើលឃើញការរីកចម្រើនគម្រោង និងគ្រប់គ្រងភារកិច្ចដែលពឹងផ្អែកលើគ្នា។

ដោយផ្តល់នូវមុខងារគ្រប់គ្រងឯកសារដែលទូលំទូលាយនៅក្នុងវេទិកា ប្លូធានាថាបទពិសោធន៍របស់អ្នកមានភាពរលូន និងមានភាពស្រួលកាន់តែខ្ពស់ បង្កើនផលិតភាព និងកាត់បន្ថយភាពរារាំង។ សម្រាប់ក្រុមដែលគ្រប់គ្រងឯកសារច្រើន ឬឯកសារធំៗ វិធីសាស្ត្ររបស់ប្លូផ្តល់អត្ថប្រយោជន៍ច្បាស់។ សមត្ថភាពក្នុងការគ្រប់គ្រងឯកសារធំៗ រៀបចំវាឱ្យមានប្រសិទ្ធភាព និងចែករំលែកវាឱ្យមានភាពរលូនអាចធ្វើឱ្យមានភាពប្រសើរនៅក្នុងការងារ។ 

ក្នុងការប្រៀបធៀប កំណត់របស់អាសាណាអាចត្រូវការឧបករណ៍ និងដំណោះស្រាយបន្ថែម ដែលអាចនាំឱ្យមានការងារដែលមានភាពខុសគ្នានិងកាត់បន្ថយផលិតភាព។ សម្រាប់ក្រុមដែលផ្តល់អាទិភាពទៅលើប្រព័ន្ធគ្រប់គ្រងឯកសារដែលមានភាពរលូន និងរួមបញ្ចូល ប្លូគឺជាជម្រើសដែលមានអត្ថប្រយោជន៍សម្រាប់អាសាណា។



### ការបង្ហាញ

ទាំងអាសាណា និងប្លូមានការបង្ហាញ ប៉ុន្តែពេលដែលអ្នកចូលទៅក្នុងព័ត៌មានលម្អិត អ្នកនឹងសង្កេតឃើញថាមានកំហុសធ្ងន់ធ្ងរនៅក្នុងរបៀបដែលអាសាណាបានអនុវត្តការបង្ហាញរបស់ពួកគេ។

មួយក្នុងចំណោមអ្វីដែលត្រូវការដើម្បីបង្កើតការបង្ហាញដែលត្រឹមត្រូវ និងមានឥទ្ធិពល គឺសមត្ថភាពក្នុងការកំណត់តាម *ច្បាស់លាស់* ទិន្នន័យដែលអ្នកត្រូវការ។ អ្វីដែលកើតឡើងក្រោមនេះ អ្នកកំពុងមើលការបញ្ជូនទិន្នន័យរបស់អ្នក និងប្រើប្រាស់ប្រព័ន្ធទីបីដូចជា Metabase ឬ PowerBI។

ការបង្ហាញអាសាណាមានកំណត់ជាច្រើនពេលដែលនិយាយអំពីការកំណត់តាមវាលកំណត់ផ្ទាល់ខ្លួន។ 

ការកំណត់តាមវាលកំណត់ផ្ទាល់ខ្លួនអាចប្រើបាន ប៉ុន្តែវាលប្រភេទមិនទាំងអស់គឺគាំទ្រដូចគ្នា។ 

វាលកំណត់ផ្ទាល់ខ្លួនដែលជ្រើសរើសតែមួយអាចប្រើសម្រាប់ការកំណត់តាមវាលកំណត់ផ្ទាល់ខ្លួន **ខណៈពេលដែលវាលដែលជ្រើសរើសច្រើន, ដែលមានមូលដ្ឋានលើអត្ថបទ និងវាលកំណត់ផ្ទាល់ខ្លួនចំនួនខ្លះមានសមត្ថភាពកំណត់តិចឬគ្មានសមត្ថភាពកំណត់នៅក្នុងការបង្ហាញ**។  

បន្ថែមពីនេះ វាលកំណត់ផ្ទាល់ខ្លួនដែលមាននៅក្នុងអង្គភាពទាំងមូលតែអាចប្រើសម្រាប់ការកំណត់តាមការបង្ហាញនៅកម្រិតផែនការ ឬការបង្ហាញសកល ដែលអាចជាបញ្ហាសម្រាប់គម្រោងដែលប្រើវាលកំណត់ផ្ទាល់ខ្លួនដែលមានលក្ខណៈជាក់លាក់។

បន្ថែមពីនេះ មានកំណត់ក្នុងការប្រើប្រាស់លក្ខខណ្ឌ "ណាមួយ" ឬ "មិន" ជាមួយការកំណត់តាមវាលកំណត់ផ្ទាល់ខ្លួននៅក្នុងការបង្ហាញ ខណៈពេលដែលមិនមាននៅក្នុងការស្វែងរក។ នេះអាចធ្វើឱ្យវាពិបាកក្នុងការបង្កើតការកំណត់តាមស្មុគស្មាញ ឬការចូលរួម។ 

អ្នកប្រើក៏បានរាយការណ៍ពីភាពមិនស្ថិតស្ថេរនៅក្នុងភាពអាចប្រើប្រាស់របស់វាលកំណត់ផ្ទាល់ខ្លួនជាច្រើនក្នុងជម្រើសការកំណត់ ដោយវាលខ្លះបង្ហាញជាអ័ក្សនៅក្នុងតារាង ប៉ុន្តែគ្មានជម្រើសសម្រាប់ការកំណត់។ កំណត់ទាំងនេះអាចមានឥទ្ធិពលយ៉ាងខ្លាំងលើភាពអាចបត់បែន និងជ្រាលជ្រៅនៃការវិភាគដែលអាចធ្វើបានតាមរយៈការបង្ហាញអាសាណា។

ប្លូយកយុទ្ធសាស្ត្រដែលខុសគ្នាទាំងស្រុង។ [យើងមានការកំណត់ខ្ពស់](https://documentation.blue.cc/views/filters#advanced-filters) ដែលអាចប្រើបាននៅលើទិដ្ឋភាពគម្រោងរបស់យើង។ នៅពេលបង្កើតការបង្ហាញ អ្នកអាចប្រើប្រាស់ *ការកំណត់ដដែល*។ 


## API & Webhook

### API

អាសាណាបានផ្តល់ការចូលដំណើរការប្រព័ន្ធ API ពេញលេញដែលមានកំណត់អតិបរិមា ១,៥០០ ការស្នើសុំក្នុងមួយនាទី។ API នេះអនុញ្ញាតឱ្យអ្នកចូលដំណើរការទាំងអស់នៃចំណុចចូលរួម រួមមានចំណុចចូល "ស្វែងរក" ដែលមានសារៈសំខាន់។ ទោះយ៉ាងណា មានកំណត់ក្នុងការអានចំនួន ៥០,០០០ ធាតុក្នុងកុងតឺន័រ ដូចជា ភារកិច្ច, រឿងរ៉ាវ, ឬគម្រោង។ API ប្រើអន្តរកម្ម RESTful, គាំទ្រទិន្នន័យ JSON ឬមាតិកាដែលបានកំណត់ទម្រង់ក្នុងការស្នើសុំ និងត្រឡប់មកវិញជាទិន្នន័យ JSON។ 

អាសាណាបានផ្តល់បណ្ណាល័យអតិថិជនសម្រាប់ភាសាកម្មវិធីជាច្រើនដូចជា Node.js, Java, និង PHP ដែលអនុញ្ញាតឱ្យអ្នកអភិវឌ្ឍន៍អានព័ត៌មាន, បញ្ចូលទិន្នន័យ, និងធ្វើឱ្យប្រតិបត្តិការពាក់ព័ន្ធនឹងការផ្លាស់ប្តូរ។ សម្រាប់ការផ្ទៀងផ្ទាត់ អាសាណាប្រើ OAuth 2.0 (Asana Connect) ហើយក៏គាំទ្រទៅកាន់កូនសោចូលផ្ទាល់សម្រាប់ការចូលដំណើរការផ្ទាល់ខ្លួន និងការអភិវឌ្ឍន៍។ API មានវិធានការសុវត្ថិភាពដូចជា CORS headers, ការផ្ទៀងផ្ទាត់ហត្ថលេខាស្នើសុំ, និងការត្រួតពិនិត្យពេលវេលាដែលផុតកំណត់។

ទោះយ៉ាងណា មានការរិះគន់ខ្លះៗចំពោះ API របស់អាសាណា។ 

មុខងារខ្លះៗដូចជា ការធ្វើច្បាប់ មិនអាចចូលដំណើរការតាមរយៈ API បានទេ។ បន្ថែមពីនេះ កំណត់ API ខ្លះៗមិនបានចុះបញ្ជីយ៉ាងល្អ ដោយធ្វើឱ្យមានការលំបាកសម្រាប់អ្នកអភិវឌ្ឍន៍។ ការចូលដំណើរការភារកិច្ចរងអាចមានភាពអត់ប្រសិទ្ធភាព និងអាចនាំឱ្យមានបញ្ហាកំណត់អតិបរិមា ពីព្រោះ API មិនផ្តល់វិធីសាស្ត្រដែលត្រឹមត្រូវក្នុងការកំណត់ថាតើភារកិច្ចមានភារកិច្ចរងឬអត់ ដោយមិនត្រូវការធ្វើសំណើសុំបន្ថែមទៀត។ 

បន្ថែមពីនេះ API ខ្វះវិធីសាស្ត្រដែលមានប្រសិទ្ធភាពក្នុងការទទួលបានព័ត៌មានដូចជា "subtask_count" ឬ "attachments_count" ដោយមិនត្រូវការសំណើសុំបន្ថែម។

### Webhooks
Webhooks អនុញ្ញាតឱ្យកម្មវិធីត្រូវបានជូនដំណឹងអំពីការផ្លាស់ប្តូរនៅក្នុងអាសាណានៅពេលពិត។ ពួកវាធ្វើការជាមួយដូចជា ធនធានព្រឹត្តិការណ៍របស់អាសាណា ប៉ុន្តែ "បញ្ជូន" ព្រឹត្តិការណ៍តាមរយៈ HTTP POST ទៅកាន់ URL ដែលបានកំណត់ ជំនួសការទាមទារការស្វែងរក។

Webhooks អាចត្រូវបានបង្កើតនៅកម្រិតខុសគ្នា៖ គម្រោង, ផែនការ, ក្រុម, ឬកន្លែងធ្វើការ។ ទោះយ៉ាងណា មានកំណត់សំខាន់ៗដែលត្រូវគិតគូរពី។ ព្រឹត្តិការណ៍នៅកម្រិតភារកិច្ច រួមមានភារកិច្ចរង និងរឿងរ៉ាវ មិនអាចបន្តទៅកាន់ Webhooks នៅកម្រិតខ្ពស់ជាងនេះដូចជា Webhooks នៅកម្រិតកន្លែងធ្វើការបានទេ។ Webhooks នៅកម្រិតកន្លែងធ្វើការទាមទារឱ្យមានការកំណត់យ៉ាងហោចណាស់មួយ ហើយការកំណត់ទាំងអស់ត្រូវតែមាននៅក្នុងបញ្ជីស្វែងរករបស់អាសាណា។

Webhooks របស់អាសាណាធ្វើការត្រួតពិនិត្យសុខភាពជាប្រចាំ ដែលគេហៅថា "ព្រឹត្តិការណ៍បេះដូង" ឬ "ការត្រួតពិនិត្យសុខភាព"។ គ្មានប្រព័ន្ធជូនដំណឹងដែលមាននៅក្នុងការជូនដំណឹងអ្នកប្រើពេលដែល Webhook មិនមានសកម្មភាពដោយសារតែការត្រួតពិនិត្យសុខភាពបានបរាជ័យ។

Webhooks របស់អាសាណាផ្តល់ជូននូវជម្រើសកំណត់លម្អិត ដែលអនុញ្ញាតឱ្យអ្នកកំណត់ការជូនដំណឹងទៅតាមតម្រូវការពិសេស។ អ្នកអាចបង្កើត Webhooks សម្រាប់គម្រោងជាក់លាក់, ផែនការ, ក្រុម, ឬកន្លែងធ្វើការ ដែល Webhooks នៅកម្រិតគម្រោងត្រូវបានណែនាំសម្រាប់ព្រឹត្តិការណ៍ដែលពាក់ព័ន្ធនឹងភារកិច្ច។

ប្រព័ន្ធគាំទ្រការកំណត់ព្រឹត្តិការណ៍លម្អិតតាមរយៈវត្ថុ WebhookFilter។ នេះអនុញ្ញាតឱ្យអ្នកកំណត់ប្រភេទធនធាន (ឧ. ភារកិច្ច, រឿងរ៉ាវ) និងសកម្មភាព (បានបន្ថែម, បានផ្លាស់ប្តូរ, បានលុប) ដែលបង្កើតការជូនដំណឹង។ សម្រាប់ភារកិច្ច អ្នកអាចកំណត់តាមវាលជាក់លាក់ដូចជា ការផ្លាស់ប្តូរថ្ងៃកំណត់។

ការកំណត់ច្រើនអាចត្រូវបានបញ្ចូលក្នុងការកំណត់ Webhook មួយ ដូចជាបញ្ជីស្វែងរកដែលអាចធ្វើឱ្យមានការប្រកួតប្រជែងណាមួយដែលបង្កើតការជូនដំណឹង Webhook។ ភាពអាចបត់បែននេះអនុញ្ញាតឱ្យមានការគ្រប់គ្រងច្បាស់លាស់លើការជូនដំណឹង Webhook ពីការអាប់ដេតនៅកម្រិតគម្រោងទាំងមូលទៅកាន់ការផ្លាស់ប្តូរវាលភារកិច្ចជាក់លាក់។

ប្លូផ្តល់ការចូលដំណើរការតាមកម្មវិធី *និង* មុខងារបណ្ណាល័យសម្រាប់បង្កើត, មើល, និងគ្រប់គ្រង Webhooks។ 

ជម្រើស Webhook របស់ប្លូក៏ផ្តល់នូវវិធីសាស្ត្រដែលមានភាពល្អប្រសើរ ដោយអនុញ្ញាតឱ្យអ្នកប្រើជ្រើសរើសព្រឹត្តិការណ៍ជាក់លាក់ដើម្បីបង្កើតការជូនដំណឹង។ កម្រិតនៃការប្ដូរនេះរួមមានសកម្មភាពលើកំណត់, បញ្ជី, វាលកំណត់ផ្ទាល់ខ្លួន, ស្លាក, និងមតិយោបល់ ជាដើម។ ភាពល្អប្រសើរនេះធានាថាអ្នកអាចបង្កើត Webhooks ដើម្បីត្រូវការពិតប្រាកដរបស់អ្នក ដោយតែបង្កើតនៅលើព្រឹត្តិការណ៍ដែលមានសារៈសំខាន់បំផុត។

បន្ថែមពីនេះ Webhooks របស់ប្លូអាចត្រូវបានកំណត់តាមគម្រោងមួយឬច្រើន ទ bahkan នៅក្នុងអង្គភាពឬកន្លែងធ្វើការផ្សេងៗ។ នេះមានន័យថាអ្នកអាចបង្កើតច្បាប់ស្វ័យប្រវត្តិ និងការជូនដំណឹងដែលមានភាពស្មុគស្មាញដែលអាចធ្វើឱ្យមានការសហការនិងផលិតភាពកាន់តែខ្ពស់នៅក្នុងក្រុម និងផ្នែកផ្សេងៗ។ សមត្ថភាពក្នុងការគ្រប់គ្រងការកំណត់ទាំងនេះតាមរយៈមុខងារដែលងាយស្រួលធ្វើឱ្យដំណើរការងាយស្រួល ទោះបីជាមនុស្សដែលគ្មានជំនាញកម្មវិធីក៏ដោយ។
